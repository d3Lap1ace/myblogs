# Java 异常处理四大模式

## 用途

替代散落各处的 try-catch，用结构化模式统一处理异常，减少静默失败、提升可维护性。

## 内容

### 核心决策模型

| 情景 | 用什么 |
|---|---|
| 基础设施失败（DB 宕机、网络超时） | 抛异常 → 全局 handler |
| 业务规则冲突（余额不足、重复注册） | `Result` 对象 |
| 非法输入（格式错误、缺少字段） | 入口验证（`@Valid`） |
| 第三方 API 不可用 | 抛自定义异常 + 全局 handler |

---

### 模式 1：集中式全局异常处理器

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<ErrorResponse> handleNotFound(UserNotFoundException e) {
        return ResponseEntity.status(404).body(new ErrorResponse(e.getMessage()));
    }

    @ExceptionHandler(Exception.class)
    public ResponseEntity<ErrorResponse> handleGeneral(Exception e) {
        return ResponseEntity.status(500).body(new ErrorResponse("Internal server error"));
    }
}
```

替代每个 Controller 各写 try-catch，所有异常响应格式统一。

### 模式 2：自定义异常体系

```java
// 基类携带 HTTP 状态码
public abstract class AppException extends RuntimeException {
    private final HttpStatus status;
    public AppException(String message, HttpStatus status) {
        super(message);
        this.status = status;
    }
    public HttpStatus getStatus() { return status; }
}

// 具体异常
public class UserNotFoundException extends AppException {
    public UserNotFoundException(Long id) {
        super("User not found: " + id, HttpStatus.NOT_FOUND);
    }
}

public class BusinessRuleException extends AppException {
    public BusinessRuleException(String message) {
        super(message, HttpStatus.UNPROCESSABLE_ENTITY);
    }
}
```

### 模式 3：Result 对象（预期失败不用异常）

```java
public class Result<T> {
    private final T value;
    private final String error;
    private final boolean success;

    private Result(T value, String error, boolean success) {
        this.value = value;
        this.error = error;
        this.success = success;
    }

    public static <T> Result<T> success(T value) {
        return new Result<>(value, null, true);
    }

    public static <T> Result<T> failure(String error) {
        return new Result<>(null, error, false);
    }

    public boolean isSuccess() { return success; }
    public T getValue() { return value; }
    public String getError() { return error; }
}

// 使用
public Result<User> registerUser(String email) {
    if (userRepository.existsByEmail(email)) {
        return Result.failure("Email already registered");
    }
    return Result.success(userRepository.save(new User(email)));
}
```

### 模式 4：入口验证（先验证，再执行）

```java
public record CreateUserRequest(
    @NotBlank String name,
    @Email String email,
    @Min(18) int age
) {}

// Controller 加 @Valid 触发验证
@PostMapping("/users")
public ResponseEntity<User> createUser(@Valid @RequestBody CreateUserRequest request) {
    return ResponseEntity.ok(userService.create(request));
}

// 全局 handler 统一处理验证失败
@ExceptionHandler(MethodArgumentNotValidException.class)
public ResponseEntity<ErrorResponse> handleValidation(MethodArgumentNotValidException e) {
    String message = e.getBindingResult().getFieldErrors().stream()
        .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
        .collect(Collectors.joining(", "));
    return ResponseEntity.badRequest().body(new ErrorResponse(message));
}
```

## 备注

- **Result 模式在 Java 中偏重**，Go/Rust 原生支持；Java 17+ 可考虑用 `sealed interface` 实现更优雅的 Result
- **注意事务边界**：`@RestControllerAdvice` 捕获异常时事务已结束，不能在 handler 里做回滚
- **异常要携带上下文**：抛出时带上足够的调试信息（ID、参数值），不要只 `throw new SomeException("error")`
- **区分"意外失败"vs"预期失败"** 是关键：`UserNotFoundException` 在某些场景是预期的（用 Result），在另一些场景是意外的（用异常）
